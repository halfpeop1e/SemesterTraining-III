## 1. 统一数据对象（立项报告 §4.3）

> 系统约定三份"统一数据对象"，跨模块复用。

| 数据对象 | 内容 | 主要使用模块 |
|----------|------|--------------|
| `train_model` | 车型参数、质量、牵引、制动、供电 | 动力学、供电、停站 |
| `line_profile` | 站点、距离、限速、坡度、停车点 | 线路、动力学、停站 |
| `power_profile` | 供电制式、功率阈值、再生制动效率 | 供电能源 |

归属（与代码对照）：
- `line_profile` → 已实现为 `signal.model.LineProfile` + 17 个 model 类 + `LineProfileLoader`（黄豪负责，其他模块复用）。
- `train_model` → 车载/车辆模块负责定义（黄豪侧 `MaConfig` 是 MA 派生配置，非 train_model）。
- `power_profile` → 供电能源模块负责定义。

## 2. 通信与接口设计（立项报告 §4.2）

> "消息总线 + 协议适配器"；实时安全数据走 DDS，周期状态走 MQTT，非实时配置走 RESTful。小学期采用简化版接口设计。

| 通信场景 | 协议 | 应用 |
|----------|------|------|
| 实时安全数据 | DDS (RTPS)，发布/订阅，QoS，延迟<1ms | ATP/ATO 指令、MA 授权、位置报告 |
| 周期性状态数据 | MQTT 5.0，QoS0/1/2 | 供电状态、设备状态、环境数据 |
| 非实时/配置数据 | RESTful API (HTTP/2) | 运行图加载、场景配置、仿真控制 |
| 大数据/日志 | Kafka / gRPC Stream | 仿真过程记录、能耗数据流 |

> 本模块已暴露：`compute(...)` 纯函数（同进程直接调，最低延迟）+ `POST /api/signal/ma`（RESTful 备选）。

## 3. 三层数据结构（小学期信号解析.md）

### 3.1 第一层：静态线路数据（来自 Excel）
Seg、站台、限速、坡度、应答器、信号机、进路等。→ 已全部建模为 `signal.model.*`（黄豪）。

### 3.2 第二层：车辆参数数据（`train_model`，需自行配置）
示例（车辆模块定义，非 MA）：
```json
{
  "train_model": "A-Type",
  "max_traction_acc": 1.0,
  "service_brake_acc": 0.8,
  "emergency_brake_acc": 1.2,
  "brake_response_time": 0.8,
  "max_speed_mps": 22.2,
  "train_length_m": 120,
  "jerk_limit": 0.75
}
```
最关键三项：常用制动减速度、紧急制动减速度、制动响应时间。MA 侧经 `MaConfig.aBrakeMps2 / tReactionS` 消费其制动能力。

### 3.3 第三层：动态运行数据（列车实时状态，运行时产生）
示例（段相对 + m/s 表示）：
```json
{
  "train_id": "T001",
  "current_seg_id": 13,
  "offset_m": 82.6,
  "speed_mps": 12.5,
  "acceleration_mps2": -0.4,
  "traction_level": 0.0,
  "brake_level": 0.35,
  "mode": "ATO",
  "target_platform_id": 1
}
```
> ⚠️ MA 的 `TrainState` 用**绝对里程 m + km/h**（非段相对+m/s）。其他模块喂 MA 前用 `TrainStateAdapter.fromSegmentRuntime(...)` 统一转换，勿新建第二个 TrainState。`traction_level/brake_level/mode/target_platform_id` 等车辆内部控制量不进 MA。

## 4. 运行时事件结构（小学期信号解析.md）

| 结构 | 示例 | MA 侧现状 |
|------|------|-----------|
| 应答器经过事件 | `{train_id, balise_id, event:"BALISE_PASSED", corrected_seg_id, corrected_offset_m}` | `Balise`（静态）；事件归车辆/引擎 |
| 区段占用 | `{section_id, occupied, occupied_by}` | `AxleCounterSection` 有 `occupied`，**缺 occupied_by**（需扩展） |
| 信号机状态 | `{signal_id, aspect:"STOP", allow_pass, reason}` | `Signal.aspect:SignalAspect`（`allow_pass=aspect.isProceed()`），**缺 reason**（需扩展） |
| 进路状态 | `{route_id, route_name, status:"AVAILABLE", locked_by_train}` | `Route`（静态），**缺 status/locked_by_train**（总控维护独立 Map 或扩展） |
| 停站结果 | `{train_id, platform_id, target/actual_position_m, stop_error_m, stop_success, door_alignment}` | 归评估/车载模块 |

> 规则：区段占用/信号状态/进路状态别另建类，**扩展现有 model**（见交接文档"允许改动耦合点"）。

## 5. 704 司机台协议编码（信号系统 3.3.2 + 司机台）

### 5.1 方向编码（4 套，MA 为方向定义权威）
| 子系统 | 编码 |
|--------|------|
| 信号系统 | `0x55` 上行 / `0xaa` 下行 / `0xff` 无效 |
| 司机台信号屏 | `0` 上行 / `1` 下行 |
| 视景 | `+1` 正 / `-1` 反 |
| 车辆 | `float` 方向 |

> MA 提供 `Direction.fromDriverConsole(int)`（0→UP/1→DOWN/其余→INVALID）；`INVALID` 触发 fail-safe。

### 5.2 信号机显示编码（文档内共 3 套）
- **3.3.2「信号系统⇒总控」（MA 权威）**：`0x01`红灯 / `0x02`黄灯 / `0x03`红黄灯 / `0x04`绿灯 / `0x05`黄灭 / `0x06`红灭 / `0x07`绿灭 / `0x08`白灯 / `0x0A`蓝灯 / `0x09`红断 / `0x10`绿断 / `0x20`黄断 / `0x30`白断。
- 视景 TCMS2VIEW：`0x01`红 / `0x02`绿 / `0x03`白。
- 视景表2：`0x00`无效 / `0x01`红 / `0x02`绿 / `0x04`白 / `0x10`黄 / `0x40`蓝 / `0x11`红黄。

> MA 用 `SignalAspect.decodeSignalSystem(int)` 解码 3.3.2；`toDriverConsoleScreen()` 聚合到司机台 红/绿/白。另两套由总控/视景各自解码，不进 MA。

### 5.3 道岔状态编码
`0x01` 定位 / `0x02` 反位 / `0x04` 四开。→ `SwitchState{NORMAL, REVERSE}`（四开暂以"非 NORMAL"覆盖 fail-safe，需要时扩 `FAIL` 枚举）。

### 5.4 列车状态字段（704 信号系统 3.3.1 输入）
列车ID、速度（cm/s）、积累走行距离（cm）、方向、长度、载重、故障限速（cm/s）、定位丢失、完整性状态、可用制动数量。

> MA 的 `TrainState` 当前含前 5 项基础字段；载重/故障限速/定位丢失/完整性/可用制动数量属车辆/ATP 输入，按用户边界暂未做（见对齐清单）。
> "积累走行距离(cm)"坐标系待周锦泽确认（是否=绝对里程×100、原点=起点道岔），确认后补 `TrainStateAdapter` 入口。

## 6. 时效与周期（704）
总控→信号 100ms；信号→总控 250ms；ATP 160ms；司机台 PLC/HMI 100ms；视景 100ms；车辆 UDP 20ms；ATP 报文时效 **1.6s**。
> MA `maValiditySec` 当前默认 5.0s（偏松），与 1.6s 时效的对齐待联调按 tick 频率定（见对齐清单 D3）。
