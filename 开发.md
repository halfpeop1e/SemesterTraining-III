# 轨道交通联合仿真平台开发详细指导

## 一、平台核心架构细化

### 1.1 三层架构落地建议

录音和需求文档中描述的架构，可以落地为以下具体技术实现方案：

**第一层：控制管理层**

这一层是平台的操作入口和监控中心，需要重点解决的是“如何让多个服务协同工作”的问题。

- **远程访问系统**：可采用基于Web的架构，用户通过浏览器访问平台。开发时需要考虑用户权限分级（如管理员、操作员、观察员），不同角色可见的操作界面和数据范围不同。实际应用中，这类系统需要实现用户登录、权限管理、远程控制平台运行状态等功能。
- **一体化监控软件**：这是平台的“大脑”，负责启动/停止各仿真子系统、配置运行参数、监控数据流、报警等。开发时建议采用微服务架构思想，每个仿真子系统作为一个独立服务，由监控软件统一调度。轨道交通仿真平台通常由主控机、数据库机和仿真机组成分布式结构，通过网络通信（如TCP/IP协议）实现互联。
- **一体化可视化系统**：负责三维视景、信号状态、供电状态等的统一展示。可采用WebGL/Three.js/Cesium等技术实现浏览器端渲染，也可采用专门的视景仿真引擎。

**第二层：实时数据库（数据总线）**

这是平台最关键的环节，所有子系统通过它交换数据。录音中提到“司机台直接去唐山服务器的IP拉数据”，说明当前采用中心化数据交换模式。开发时推荐：

- **轻量级方案**：采用Redis作为实时数据总线，各子系统通过发布/订阅模式交换数据。Redis的高性能和丰富数据结构非常适合轨道交通仿真场景。
- **企业级方案**：采用DDS（数据分发服务），它是工业物联网和仿真领域常用的数据总线标准，支持QoS配置，能满足轨道交通对实时性和可靠性的要求。研究案例表明，分布式异构数据库同步可以使用基于“映射文件”的方法，采用XML作为中间数据表示方式，有效避免异构数据库之间标准不统一的问题。
- **数据同步机制**：各子系统向数据库写入数据，其他子系统从数据库读取。需要设计清晰的数据表结构和更新机制。专用数据库通常需要区分动态数据（如列车位置、速度）和静态数据（如线路参数、设备配置）两类。

**第三层：业务仿真层**

各专业子系统在这一层运行。结合录音和需求文档，模块划分可进一步明确：

| 模块名称 | 核心职责 | 开发建议 |
|---------|---------|---------|
| 供电仿真模块 | 模拟DC1500V牵引供电、能馈、储能 | 输出网压、网流、功率等参数给车辆模型 |
| 线路仿真模块 | 提供线路、坡度、曲线、限速、站台信息 | 采用有向图描述线路拓扑，便于路径搜索和位置计算 |
| 虚拟列车模块 | 含网络、牵引、制动、动力学四大子系统 | 四大子系统内部闭环，对外提供速度/位置/加速度 |
| 信号仿真模块 | 模拟ATS、ATO、ATP、联锁、CBTC | 根据车辆位置和进路状态控制行车许可 |
| 半实物司机台 | 连接真实司机台设备 | 通过协议转换将物理操作转为数字信号 |

### 1.2 网络与通信规划

录音中提到“机器IP都是手动设的固定IP”，开发团队需要提前规划IP分配表：

- **网段规划**：统一使用一个私有网段（如192.168.x.0/24），避免IP冲突
- **IP分配表**：为每台机器（服务器、司机台电脑、信号机、视景机等）分配固定IP
- **端口规划**：每个子系统开放特定端口，避免端口冲突。录音中提到司机台内部三个设备走了三个端口，说明同一台机器可能需要监听多个端口对应不同功能
- **交换机连接**：如果设备数量较多，建议采用级联交换机方式；如果只是开发测试，可以用小型交换机临时组网

## 二、各子系统实现指导

### 2.1 虚拟列车模块

虚拟列车是核心仿真对象，由四个子系统组成，其数据流关系在需求文档中有清晰描述。

**动力学子系统**：

这是最基础的模块，负责计算列车运动状态。录音中提到“加速度就可以有根据时间计算出速度”，开发建议：

- 输入：牵引力、制动力、坡度、曲线阻力、列车质量
- 输出：速度、位置、加速度
- 核心算法：根据牛顿第二定律 F=ma，结合列车阻力公式计算实时加速度，再通过时间积分得到速度和位移
- 进阶实现：支持质量可变（空载/满载/超载）、故障模拟（如部分动力丢失）
- 仿真步长：建议使用固定步长（如50ms或100ms），便于各模块间数据同步

**牵引与制动子系统**：

- 牵引系统：根据牵引指令百分比计算实际牵引力输出，需考虑电机特性曲线
- 制动系统：包含电制动和空气制动，需协调两者关系（电制动优先，不足时补充空气制动）
- 两者之间直接交换电制动力请求/释放信息，实现协调控制

**网络（TCMS）子系统**：

负责列车内部各设备通信状态监控，开发时可简化为状态上报和故障注入接口。

### 2.2 信号仿真模块

信号系统是控制列车运行的核心，包含：

- **ATS（自动列车监控）** ：负责行车调度、进路控制、运行图管理
- **ATP（自动列车保护）** ：负责超速防护、安全制动距离计算，是安全核心
- **ATO（自动列车运行）** ：负责自动驾驶、精准停车
- **联锁**：负责道岔、信号机、轨道区段的联锁逻辑

开发建议：录音中提到需要先“加车”让信号系统识别新车辆，然后设置交路驾驶模式，方可交由自动驾驶。这一过程需要实现：

1. 车辆注册：信号系统与车辆建立通信，获取车辆ID、位置、状态
2. 交路设置：配置列车运行路径（始发站、终点站、停站安排）
3. 驾驶模式切换：手动驾驶→自动驾驶的状态转移逻辑
4. 精准停车控制：需求中提到停车精度需控制在10cm以内，否则车门无法打开，说明ATO算法需要高精度

联锁仿真平台可以采用SQL数据库提供的数据存储环境，使用面向对象语言开发，根据线路信息编写数据库，利用线路设备静态数据绘制界面。

### 2.3 半实物司机台接口

录音中详细描述了司机台的连接方式：司机台后有一台电脑，运行通信程序和场景程序，从服务器拉取数据并显示。开发要点：

- **硬件接口**：司机台内部有多个设备（板卡、部件），通过三个端口与外部通信
- **协议适配**：需要开发适配程序，将司机台的物理操作（手柄、按钮）转换为数字信号，发送给实时数据库
- **数据显示**：司机台显示器需展示速度表、距离、车站信息、牵引/制动状态等
- 实际列车控制仿真系统通常包括列车控制操作台（含操控面板、信号系统显示屏、轨道模拟显示屏、司机台显示器）、列车控制管理系统模拟器和环境模拟装置

### 2.4 供电仿真模块

录音中专门问了“供电仿真是做啥的”，说明这一模块容易被忽略，但它是车辆运行的基础支撑：

- 模拟牵引变电站、接触网、能馈装置
- 输出参数：网压、网流、功率、能耗
- 与车辆模型的交互：车辆取电行为影响供电网络状态，供电状态反过来限制车辆性能
- 开发建议：可简化为根据车辆位置和取电功率计算供电网络压降和电流，输出给显示界面
- 目前供电仿真与车辆模型在同一个仿真机上运行，暂未对外开放直接调控接口

### 2.5 三维视景模块

负责展示三维列车运行、线路、信号状态等。开发建议：

- 使用Unity3D或Unreal Engine，或者Web方案（Three.js/Cesium）
- 需要接收实时数据库中的列车位置、速度、信号状态数据
- 与司机台显示屏形成互补：司机台显示仪表数据，视景显示外部环境
- 支持视角切换（司机室视角、车外视角、线路全景）

## 三、开发流程与调试策略

### 3.1 推荐开发顺序

根据录音中杨工的建议，建议按以下顺序分步实施：

**第一步：通信协议实现**

“协议其实就是一个相当于字节处理的一个过程” —— 这是最基础的工作。需要：

- 研读《轨交多系统平台接口协议汇总20260630》，明确每个接口的数据格式（字节序、数据类型、字段定义）
- 开发协议编解码库，供各模块调用
- 编写单元测试验证协议解析正确性
- 注意：车辆模型接口不仅是简单通信协议，还包括API调用，需在特定软件环境下测试

**第二步：简单闭环验证**

如录音建议：“显示的加司机台，加上你们自己的运行的那块”。这是最快验证方案：

1. 司机台发送操作指令（牵引/制动）
2. 你们的程序根据指令计算速度/位置/加速度
3. 视景显示列车运动
4. 这样就能看到“车在动”的效果

实现方法：在程序内部做一个查表，不同牵引指令对应不同的加速度，积分得到速度和位移。

**第三步：逐步替换**

完成单个模块的开发后，将其接入完整系统：

- 替换原有的对应模块（如用自己的车辆模型替换原来的）
- 保持其他模块不变，验证自己的程序是否正确
- 渐进式替换所有模块

### 3.2 调试要点

- **数据可视化**：开发调试界面，实时显示各模块输入输出数据，便于定位问题
- **日志记录**：记录每个模块收发数据的时间戳和内容，便于事后分析
- **逐模块验证**：先独立测试每个模块，再集成测试
- **注意车辆模型特殊性**：车辆模型的API接口需要唐山服务器上的特定软件环境，需提前准备
- 协同仿真平台往往涉及多个专业软件的联合仿真（如Fluent、Simpack、Simulink），开发时需注意各软件间的数据交互接口

## 四、进阶功能扩展

### 4.1 故障注入

开发支持故障模拟：

- 车辆故障：电机故障、制动故障、通信故障
- 信号故障：ATP失效、联锁异常
- 供电故障：接触网失电、变电站故障
- 线路故障：道岔失表示、轨道电路异常

可以通过多车控制台进行故障注入配置。

### 4.2 数据回放

录音中提到运行数据库同步系统，可用于：

- 记录所有仿真数据
- 事后回放分析
- 作为案例库用于培训和故障分析

### 4.3 多车运行

支持多列车同时在线路上运行，需要处理：

- 列车间的安全间隔
- 进路冲突
- 追踪运行

## 五、开发注意事项

1. **先读协议再写代码**：通信协议是各模块间对话的语言，理解透彻后再开发，避免返工
2. **注意接口的同步/异步特性**：部分接口需要实时响应（如安全相关），部分可以容忍延迟（如数据显示）
3. **车辆模型接口需要特定环境**：录音提到车辆模型的API需要唐山服务器上的软件环境，提前确认环境准备
4. **IP和端口规划先行**：避免开发后期因IP冲突导致联调困难
5. **采用增量开发**：不要试图一次性完成所有模块，每一步都可运行、可验证是成功的关键
6. **关注精度要求**：停车精度10cm、数据交换单位厘米等指标需要在编码时特别注意
7. **合理利用中间件**：可以使用成熟的中间件（如Redis、DDS、ZeroMQ）作为数据总线，避免重复造轮子
