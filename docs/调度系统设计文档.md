# 轨道交通调度仿真系统 — 总控调度与多车协同模块设计文档

> **成员一负责模块** | 北京地铁9号线 | Spring Boot + React

---

## 系统定位声明

本系统是基于 CBTC / ATS / ATO / ATP 分层思想的**教学简化调度仿真模型**，不声称还原真实工业级 CBTC 系统，也不实现完整联锁逻辑、二进制通信协议和安全认证机制。当前阶段重点是还原以下核心调度逻辑：

1. 北京地铁9号线多车运行
2. 运行图与发车间隔
3. 客流驱动的发车间隔调整
4. 多车追踪与安全间隔
5. 站停、折返、晚点传播
6. 总控与车载系统的数据闭环
7. 供电区段能耗与峰值风险
8. 为后续对接实验室接口协议预留 ProtocolAdapter

由于无法直接获得9号线真实AFC断面客流数据，本项目采用公开运营间隔和车辆定员反推高峰断面运能，并将其作为仿真参数。

## 一、系统总览与架构分层

本调度仿真系统模拟北京地铁9号线（郭公庄—国家图书馆，13站，约16km）的列车运行全过程。系统按照真实CBTC（基于通信的列车控制）系统的分层架构设计，每一层对应实际运营中的子系统职责：

| 仿真层 | 对应真实系统 | 核心职责 |
|--------|-------------|---------|
| **Layer 0: 客流模型** | 清分中心 + OD预测 | 基于时段和站点权重的断面客流计算，驱动发车间隔决策 |
| **Layer 1: 时刻表管理** | ATS 列车自动监控 | 生成全线列车计划时刻表，包含到站/发车/停站/折返时刻 |
| **Layer 2: 安全防护** | ATP 列车自动保护 | 移动闭塞安全间隔计算、移动授权（MA）生成、紧急制动触发 |
| **Layer 3: 运行控制** | ATO 列车自动驾驶 | 八状态运行状态机、运动学更新、站停管理、折返逻辑 |
| **Layer 4: 调度策略** | 调度中心人工/自动决策 | 多级赶点恢复、前后车协同优化、晚点传播控制 |
| **Layer 5: 能源优化** | 供电调度 + 节能策略 | 错峰启动、惰行窗口、再生制动协同、峰值功率控制 |

仿真引擎以1仿真秒为步长进行离散推进，每步依次执行Layer 1-5的逻辑。在各层之间，通过一个中心化的列车状态Map（trainId → TrainState）共享数据。多列车在同一个时间步内依次完成运动学更新，互不干扰；只有在需要安全检查和传播评估时才进行列车间交互。

## 二、客流模型（Layer 0）

### 2.1 时段倍率体系

客流需求不是定值，而是随一天中的时段剧烈波动。系统将全天划分为五个运营时段，每个时段有独立的客流倍率：

- **凌晨（0-6点）**：倍率 0.3，客流稀疏
- **早高峰（7-9点）**：倍率 2.0，全天客流最高峰
- **平峰（10-16点）**：倍率 0.6，正常运营
- **晚高峰（17-19点）**：倍率 1.8，次高峰
- **夜间（20-23点）**：倍率 0.4，逐渐下降

### 2.2 站点权重与上下车模型

13个车站各有不同的"重要度权重"，反映该站在路网中的地位。例如北京西站（枢纽站）权重2.0，军事博物馆（换乘1号线）权重1.4，普通中间站权重0.6-1.0。

每个站同时定义了**上客率（boardingRatio）**和**下客率（alightingRatio）**，两者之和恒为1.0。上客率从郭公庄（0.95）递减到国家图书馆（0.10），下客率则反向递增。这模拟了早高峰通勤客流的基本特征：乘客从南端住宅区上车，在市中心和北端工作区下车。

### 2.3 断面客流累加模型

这是整个客流模型的核心计算逻辑。定义第i站离开后的断面客流为：

```
SectionLoad(i) = SectionLoad(i-1) + Boarding(i) - Alighting(i)
```

其中每个站的上车量 = 基准流量(800人次/h) × 时段倍率 × 站点权重 × 上客率，下车量同理。这个累加链条从郭公庄一路推进到国家图书馆，得到全线12个运行区间的断面客流。找到其中的最大值，就是"最大断面客流"。

### 2.4 客流分类与发车间隔决策

**核心区分**：系统明确区分三类客流——

- **站点客流（stationPassengerFlow）**：某站某时段的进出站压力，反映该站的局部上下车需求
- **断面客流（sectionPassengerFlow）**：列车离开某站后、对应运行区间的车上人数，由断面累加模型计算
- **最大断面客流（peakSectionPassengerFlow）**：全线所有断面客流的最大值，是计算发车间隔的唯一依据

**发车间隔公式**：

```
h = (3600 × trainCapacity × targetLoadFactor) / peakSectionPassengerFlow
```

其中 6B 编组列车定员 1458/1460 人，目标满载率 85%。

2500 人/小时可以作为某个枢纽站的局部站点客流，但不能直接作为最大断面客流。要模拟 120 秒高峰间隔，最大断面客流应采用 30000-40000 人/小时/方向量级，或明确声明为仿真参数。凌晨低客流时段自动将间隔拉大到 500-600 秒。

**上线列车数约束**：

```
N_required = ceil(T_cycle / headway)
T_cycle = 2 × oneWayRunTime + terminalDwellAndTurnbackTime
```

如果 N_required > N_available（可用列车数 12 列），系统：
- 不允许强行使用目标发车间隔
- 输出"车辆资源不足"调度事件
- 给出建议：加开备用车、调整交路、限流、延长实际发车间隔

## 三、时刻表系统（Layer 1）

### 3.1 时刻表生成逻辑

时刻表是调度系统的"基准线"，所有晚点检测都依赖它。生成过程分两步：

**第一步：预计算各区间运行时分。** 不是简单用距离除以速度，而是按照三段式运行曲线模型——加速段（从0到巡航速度）、巡航段（匀速）、制动段（减速到0）。先计算加速到巡航速度70km/h所需的时间和距离，再减去进站制动所需的时间和距离，中间剩余距离以巡航速度通过。对于站间距离特别短的区间（如某些仅1km左右的区间），列车甚至来不及达到巡航速度就要开始制动，此时用纯加减速模型计算。

**第二步：为每列车逐一生成全站时刻表。** 列车按编号顺序以发车间隔错开发车。第一列车在t=0从郭公庄发车（始发站），之后每经过一个站，累加该区间的运行时分+本站的计划停站时间，得到该站的计划到站和计划发车时刻。第N列车则延迟(N-1)×间隔秒数发车，但各区间运行时分和停站时间相同。到达国家图书馆（终端站）后计划停站为0——因为列车在此折返，不执行常规站停。

### 3.2 计划停站时间模型

并非所有站的停站时间都一样。系统建立了一个受多重因素影响的停站公式：

```
D(i, t) = D_base + α×高峰附加 + β×枢纽附加 + γ×拥挤附加
```

基准停站时间30秒。在早晚高峰时段自动叠加15秒（上下车人流量大）。如果该站是枢纽站（如北京西站，权重≥1.8），额外加20秒。换乘站（六里桥、军博、国图，权重≥1.3）加15秒。在高峰期间的大站（权重≥1.2）还会叠加10秒拥挤附加。所有这些附加可叠加，但最终结果被限制在15秒-90秒之间。

### 3.3 车次号规则

每列车有正式的车次号，遵循城轨命名惯例：线路号（2位）+ 方向码（1=上行 2=下行）+ 序号（2位）。例如90101表示9号线上行第01次列车。方向码在折返后自动切换。

## 四、列车运行状态机（Layer 3 - ATO）

### 4.1 八状态定义与转移

列车在整个生命周期中经历九个状态（含终端站乘降清客），构成一个闭环状态机：

```
DEPOT_WAITING → DEPARTING → ACCELERATING → CRUISING → BRAKING → DWELLING
      ↑                                                               ↓
      │                                              (非终端站发车) ←─┘
      │                                                               ↓
      │                                          TERMINAL_DWELL ←─(终端站到站)
      │                                               ↓
      └──────────────────(折返完成,反方向)── TURNING_BACK ←──────────┘
                                                                      ↓
                                                                  FINISHED
```

**DEPOT_WAITING（车辆段待发）**：列车尚未上线，等待ATS根据时刻表下达出发指令。在此期间延迟会累积。

**DEPARTING（起动）**：刚收到出发指令，速度从0开始增加。当速度超过5km/h后自动转入ACCELERATING。

**ACCELERATING（加速）**：列车以固定加速度（正常模式1.0 m/s²）持续加速，直到达到目标巡航速度。

**CRUISING（巡航）**：维持恒定巡航速度（默认70km/h）运行，直到接近下一站需要开始制动。

**BRAKING（制动）**：当距离下一站台的距离 ≤ 当前速度对应的制动距离 × 1.05 时，自动进入制动阶段。制动减速度由运行等级决定（正常模式1.0 m/s²）。速度降到0且距离站台20米内时，判定为到站，转入DWELLING。

**DWELLING（站停）**：在站台停靠，计数器每秒递增实际停站时间。当实际停站 ≥ 计划停站时间时，判断下一站方向并转入DEPARTING（非终端站）或TERMINAL_DWELL（终端站）。

**TERMINAL_DWELL（终端站乘降清客）**：仅在终端站（郭公庄或国家图书馆）触发。列车停靠60秒进行乘降、清客和安全确认。完成后转入TURNING_BACK。

**TURNING_BACK（折返换端）**：折返时间根据时段动态计算——平峰180秒，高峰240秒。折返完成后切换方向（上行↔下行）、更新车次号和新tripId、清空旧调度指令，以DEPARTING状态反方向发车。

**FINISHED（终到）**：列车完成全部运行任务（可能经过了多次折返后），退出仿真。

### 4.2 线路占用规则

以下状态**占用线路轨道**，参与安全间隔、移动授权和区段占用计算：
RUNNING, ACCELERATING, CRUISING, BRAKING, DWELLING, TERMINAL_DWELL, TURNING_BACK, HOLDING, EMERGENCY_STOPPED。

不参与追踪的状态仅包括：DEPOT_WAITING, FINISHED。

静止列车（DWELLING, TERMINAL_DWELL）虽然不进行运动学更新，但仍然占用区段、是后车必须防护的对象，不能被安全间隔计算跳过。

### 4.2 运动学更新核心逻辑

每仿真秒，每列非DWELLING/FINISHED列车执行以下运动学更新：

1. **检查紧急制动状态**：如果紧急制动标志为真，强制以紧急制动减速度（1.3 m/s²）减速，速度到0后转入DWELLING。

2. **读取运行等级参数**：从当前列车的operationLevel获取巡航速度、加速度、减速度。例如节能模式55km/h、快车模式75km/h。

3. **读取赶点指令**：如果当前列车有活跃的SPEED_UP指令，根据其编码的恢复等级从恢复速度增量表中查找对应提速值，叠加到巡航速度上。例如L2级别（中度赶点）在当前巡航速度上+5km/h。

4. **甩站判定**：如果skipNextStation为真且位置已越过下一站台50米，标记甩站完成，跳到再下一站为目标站，同时记录甩站指令到日志。

5. **制动判断（非甩站时）**：计算到下一站台的距离。如果距离 ≤ 制动距离×1.05（或小于100米的安全兜底），转入BRAKING状态，用精确的运动学公式计算所需的减速度。

6. **正常加速/巡航**：如果未触发制动，继续按加速度加速到目标速度，或维持巡航。

### 4.3 方向感知与双向运行

系统完整实现了上行和下行两个方向的运行。上行（郭公庄→国家图书馆）位置递增、站索引递增，下行则相反。位移计算直接使用方向系数：上行时位移=+速度×时间，下行时位移=-速度×时间。所有与站台位置相关的计算（制动距离、甩站判定、到站检测、当前站索引更新）都考虑了方向因素。

这带来了一个重要的设计选择：下行时列车位置（公里标）递减，这符合铁路实际——公里标从郭公庄（0km）递增到国图（约16km），下行列车从国图出发时处于16km处，随运行逐渐减小到0km。

## 五、ATP安全防护系统（Layer 2）

### 5.1 移动闭塞模型（方向感知）

系统采用 CBTC 移动闭塞。核心概念是**移动授权（Movement Authority, MA）**——后车被允许行驶的最远位置点。MVP 阶段使用方向感知简化模型：

```
directionSign: UP=+1, DOWN=-1
frontTailPosition = frontTrain.positionMeters - directionSign × trainLengthMeters
MA_end = frontTailPosition - directionSign × safetyMarginMeters
rearEmergencyStopPosition = rearTrain.positionMeters + directionSign × rearBrakeDistanceMeters
```

比较规则（方向感知）：
- UP 方向：rearEmergencyStopPosition > MA_end → 越过授权 → 紧急制动
- DOWN 方向：rearEmergencyStopPosition < MA_end → 越过授权 → 紧急制动

### 5.2 参与安全间隔的状态

**DWELLING / TERMINAL_DWELL / TURNING_BACK 列车占用线路并参与 MA 计算**。停站列车虽然不进行运动学更新，但仍占用站台区段，是后车必须防护的对象。参与安全间隔和轨道占用计算的状态包括：RUNNING, ACCELERATING, CRUISING, BRAKING, DWELLING, TERMINAL_DWELL, TURNING_BACK, HOLDING, EMERGENCY_STOPPED。不参与追踪的仅包括：DEPOT_WAITING, FINISHED。

### 5.3 紧急制动触发条件

当后车的"紧急制动停车位置"（当前位置 + 后车EB距离）≥ MA时，意味着后车即使立即施加紧急制动也无法在前车尾部安全距离外停下——系统立即触发紧急制动。紧急制动一直持续到速度归零（转入DWELLING）或危险解除。

### 5.3 安全间隔预警

在触发紧急制动之前，系统有更温和的预警机制。通过比较实际间距与安全距离（制动距离 + 反应距离 + 安全余量 + 列车长度）：

- **间距 < 安全距×0.4**：紧急扣车（HOLD指令），后车完全停住
- **间距 < 安全距×0.5**：危险级别，触发SLOW指令大幅限速  
- **间距 < 安全距×0.8**：警告级别，触发SLOW指令限速
- **间距 < 安全距**：关注级别，持续监控但不采取措施
- **间距 ≥ 安全距**：安全，清除所有限速指令

反应距离考虑了系统通信延迟约2.5秒。

### 5.4 间距计算约束

在计算前后车间距时，系统会对所有活跃列车按位置从小到大排序，只比较相邻列车对。处于DWELLING状态的列车被跳过——静止列车不构成追尾威胁。此外，只有当后车与领航车处于相同方向时，常规间距计算才有意义。

## 六、调度策略体系（Layer 4）

### 6.1 调度模式决策

基于客流模型，每10仿真秒（可配置）评估一次全局调度模式：

- **NORMAL（正常运营）**：需求间隔在120-600秒正常范围内，车队运力充足
- **COMPRESS（压缩间隔）**：需求间隔小于120秒（客流过大），系统将间隔强行压至120秒的最小值，进入高密度运行
- **STRETCH（扩大间隔）**：需求间隔超过600秒（极端低客流），按最大600秒间隔运行
- **EMERGENCY（紧急）**：所需列车数超过可用列车数（12列），车不够，需要外借或压缩交路

调度模式会影响列车的运行等级推荐和发车频率调整。

### 6.2 多级赶点恢复模型（核心策略）

这是本系统最有特色的设计。不是简单地"晚点了就通知赶点"，而是根据晚点严重程度划分四个恢复等级，每个等级采用不同的恢复强度组合。

**等级判定规则**：

| 等级 | 晚点范围 | 标签 | 核心策略 |
|------|---------|------|---------|
| L1 轻度 | 15-60秒 | 缩停站 | 每个后续站停缩短15秒 |
| L2 中度 | 60-180秒 | 缩停站+提速 | 每站缩短20秒 + 巡航速度临时提至75km/h |
| L3 重度 | 180-360秒 | 缩停站+提速+建议甩站 | 每站缩短25秒 + 提至78km/h + 生成甩站建议(需人工确认) |
| L4 严重 | >360秒 | 全力恢复 | 每站缩短30秒 + 提至80km/h + 生成甩站/小交路/清客等应急建议 |

**关键设计**：恢复等级以整数0-3编码在SPEED_UP指令的targetValue字段中携带。L3/L4级别**不再自动强制甩站**，而是生成 SUGGEST_SKIP 类型的调度建议，状态为 PENDING，只有调度员通过前端手动确认后才能真正执行 SKIP_STATION / SHORT_TURN 等强干预策略。

**停站缩短硬约束**：缩短后不得低于最小停站时间。普通站 minDwell = 20s，换乘/枢纽站 minDwell = 35s，终端站 minDwell = 60s。使用公式：`newDwell = max(plannedDwell - recoveryCut, minDwell)`。

**SPEED_UP指令生命周期**：每30仿真秒进行一次全量晚点评估。如果某列车本轮不再收到新的SPEED_UP指令（说明晚点已好转到阈值以下），上一轮的指令自动被清除——列车恢复正常运行参数。

### 6.3 前后车协同优化模型（创新设计）

传统的晚点传播处理逻辑存在"误伤"问题：前车一旦晚点，系统无条件给后车减速或扣车，即使前车完全有能力自行恢复。这导致后车被"无辜牵连"，降低全线效率和乘客体验。

本系统的协同优化模型引入了**恢复潜力评估**机制：

**恢复潜力计算公式**：

```
恢复潜力 = Σ(后续各站停站缩短) + Σ(各区间提速节省) + 甩站节省
```

具体包括三个来源：
1. **停站缩短**：剩余停站数 × 该等级的缩站秒数
2. **提速节省**：剩余距离从原速(70km/h)提升到(70+增速)km/h后，在纯巡航段可以节省的时间
3. **甩站节省**：L3级别跳过约1/3非枢纽站（含停站+部分运行时间），L4级别跳过约1/2

**决策逻辑**：

- 前车的恢复潜力 ≥ 前车当前晚点的70% → 前车大概率能自行恢复，后车仅接受**轻度限速**（限速到当前速度的80%，且不低于45km/h）。后车基本不受影响，只是稍慢一点

- 前车的恢复潜力 < 前车当前晚点的70% → 前车恢复无望，后车必须**强力限速**（限速到当前速度的50%，至少25km/h），避免追尾和晚点传播链的形成

这个70%的阈值是核心调参参数——设得太高（如90%），会让系统过于乐观，导致本应控制的晚点没有及时控制；设得太低（如50%），又会让系统过于保守，频繁误伤后车。

### 6.4 晚点检测与事件管理

晚点检测的基准来自时刻表：比较列车实际到达某站的时刻与计划到站时刻。偏差超过60秒标记为初始晚点事件（PRIMARY_DELAY），传播到晚点的后车标记为传播晚点事件（PROPAGATED），恢复后偏差小于15秒标记为已恢复（RECOVERED）。

晚点事件去重存储：同一列车同一类型的事件会被最新的一条覆盖，避免日志膨胀。

## 七、能源优化系统（Layer 5）

### 7.1 错峰启动

多列车同时从车辆段发车会导致供电系统峰值功率飙升。系统为每列车计算一个微偏移量（0-8秒），叠加在计划发车时刻上。偏移量按列车的上车顺序循环分配：第1列偏移0秒，第2列偏移3秒，第3列偏移6秒……这样即使在同等发车间隔下，实际牵引功率峰值也被时间上分散了。

### 7.2 惰行窗口优化

当列车实际运行提前于时刻表至少8秒时，系统在巡航段插入惰行窗口。惰行时目标速度降低到55km/h（切断牵引力，靠惯性滑行），这不但节省能源，还能自然消耗掉提前量——列车虽然被允许滑行变慢，但由于"提前了"，晚几秒到站仍然是准点的。

触发惰行的前提条件：列车处于巡航或加速状态、速度不低于35km/h（避免溜停）、没有紧急制动。

### 7.3 再生制动协同

地铁列车制动时会进行再生制动——将动能转化为电能回馈电网。但电能不能储存，如果同一供电区段内没有其他列车在加速消耗电能，再生电能就白白以热能形式耗散了。

系统每20仿真秒检测一次：当有列车处于制动状态且制动功率超过200kW时，在3公里范围内寻找正在加速的列车。找到匹配对后，让加速列车恢复全速牵引（撤掉限速指令），吸收制动列车的再生电能。再生效率按65%估算。

### 7.4 峰值功率控制

系统实时监控所有列车的同时牵引功率。当同时处于加速或起动状态的列车超过4列（供电上限），对超出阈值的列车强制施加40km/h的SLOW限速，将功率峰值控制在供电系统承受范围内。

风险评估分为三级：总功率 < 阈值的90%为安全，90%-100%为警告，超过100%为危险。

### 7.5 运行等级自适应调整

在低客流时段（满载率 < 40%）或凌晨/夜间，系统自动将运行中的列车切换为节能模式——巡航速度降到55km/h，加减速更柔和（0.7m/s²），停站时间微增（倍率1.1）。高峰时段自动恢复为正常模式。

## 八、交路与折返管理

### 8.1 交路分配策略

系统支持四种交路模式：
- **全程交路（FULL）**：郭公庄↔国家图书馆，站0-12，覆盖全部13站
- **南段小交路（SHORT_S）**：郭公庄↔六里桥，站0-6，覆盖南段7站
- **北段小交路（SHORT_N）**：六里桥↔国家图书馆，站6-12，覆盖北段7站
- **快车模式（EXPRESS）**：全程但甩站通过部分非枢纽站

仿真初始化时，多数列车被分配为全程交路，最后1-2列（总数≥4时）分配为南段小交路以增加调度多样性。

### 8.2 折返流程

终端站到站 → DWELLING状态 → 停站完成 → TURNING_BACK状态（切换方向，更新车次号）→ 折返计时 → 计时到期 → DEPARTING反方向发车。折返时间根据当前时段在180秒（平峰）和240秒（高峰）之间选择。

### 8.3 调度员手动策略

前端提供按钮支持调度员手动干预：甩站（下一站通过）、快车（切换EXPRESS等级+自动甩站）、节能（切换ENERGY_SAVE等级）、慢行（SLOW等级）、恢复正常。这些策略通过API发送到后端，后端根据策略类型修改列车状态并记录指令日志。

## 九、数据记录与快照系统

### 9.1 位置采样

每5仿真秒采样一次所有活跃列车的位置（公里标）和方向，存入positionHistory列表。采样数据用于前端绘制"运行图"——横轴为时间，纵轴为公里标，每条折线为一列车的运行轨迹。

### 9.2 到站记录

每列车每次到站时记录一个StationArrival事件，包含：实际到站/发车时刻、计划到站/发车时刻（从时刻表查询）、计划与实际停站时间、偏差值。这些数据用于前端"计划-实绩偏差分析"页的详细表格和正点率统计。

### 9.3 能耗估算

在每仿真步中，对所有有速度且加速度非负（牵引状态或巡航维持）的列车进行牵引能耗累加：功率 = 牵引力（质量×加速度/3.6 + 2000N基础阻力）× 速度，再除以3600000换算为kWh。

### 9.4 快照输出

getSnapshot()方法将所有内部状态聚合为一个SimulationSnapshot对象，包含：列车列表、位置历史、到站记录、车头时距、活跃指令、指令日志、晚点事件日志、客流快照、调度决策、能耗数据、能源优化数据、计划运行图点位。前端通过轮询（每500ms一次）获取最新快照并渲染。

## 十、仿真控制流程

### 10.1 启动/暂停/继续

**首次启动**：前端调用POST /api/simulations/start，后端初始化所有列车、生成时刻表、设置仿真时间为0。返回后前端开始500ms间隔的轮询推送（调用step + snapshot）。

**停止**：前端清除interval定时器，后端仿真时钟冻结，但所有状态（trains、positionHistory、stationArrivalMap等）保留在内存中。

**继续**：用户再次点击"启动"按钮时，后端检测到trains Map非空（说明之前已启动过），仅将simulationRunning设为true并直接返回，不重新初始化。前端的轮询恢复后，仿真从停止时刻无缝继续推进。这是关键的"避免重新开始"机制——关键在于后端不重新执行清空和初始化逻辑。

### 10.2 重置

重置调用POST /api/simulations/reset，后端将一切归零：清空列车Map、清空所有历史记录、清空指令和事件日志、释放能源优化缓存。前端也同步清空快照状态。之后用户可重新启动全新的仿真。

### 10.3 仿真时钟权威控制

**后端是仿真时钟的唯一权威**。前端只负责发送 `start / pause / reset / strategy / manualCommand` 指令，通过 snapshot 轮询读取状态。为防止多个客户端同时打开（中控页 + 车载页）重复推进仿真时间，后端增加了最小步进间隔检查（300ms 窗口期内重复请求被忽略）。如果短期无法重构为 SSE/WebSocket，至少确保只有 control-center 可以推进 step，onboard-system 只能读取状态和上报状态。

## 十四、总控系统与车载系统协同交互设计

### 13.1 现状与目标

**当前状态**：总控调度系统（成员一）和车载仿真系统（成员三 郭逸晨）是两个独立运行的系统。总控侧运行多列车全线步进仿真，生成调度指令；车载侧运行单列车单区间纵向动力学仿真，输出精确的速度-位置曲线和停站精度。两者之间没有任何数据交换——这是一种"开环"状态。

**目标**：建立闭环协同机制。总控系统作为"大脑"，观察全线列车状态后下发调度指令；车载系统作为"执行终端"，接收指令后在本地调整运行参数并执行，同时将实际运行状态、安全事件和异常情况回传给总控，总控据此重新评估全线态势并调整调度策略。这种双向闭环是真实城轨CBTC系统（基于通信的列车控制）的核心工作模式。

### 13.2 整体交互架构

│        │              │              │              │       │

│                       ▼              ▼                      │

总控与各车载系统之间不是网状直连，而是通过一个逻辑上的**指令/状态总线**进行通信。每个车载系统只需要与总控通信，不直接与其他车载系统交互。

### 13.3 通信方式：轮询 + 事件驱动混合

考虑到本项目的规模和复杂度，采用**混合模式**：

**下行通道（总控 → 车载）：轮询拉取**

车载系统以固定频率（建议每500ms，与总控仿真步进同步）向总控查询是否有属于自己的新指令。原因是调度指令由总控每30秒集中评估一次后生成，频率不高，轮询足够且实现简单。

**上行通道（车载 → 总控）：混合模式**

- **周期性状态上报**：车载系统每500ms（与总控仿真步进同步）将自己的运行状态推送给总控，包括位置、速度、加速度、运行阶段等。总控用这些数据更新自己的列车状态视图，确保调度决策基于最新真实状态。

- **事件驱动即时上报**：车载系统在检测到须立即响应的事件时（ATP紧急制动触发、故障检测、异常停站偏差），不等下一个轮询周期，立即推送事件通知到总控。这样总控可以在最短时间内（一个仿真步长的延迟）重新评估全线调度。

### 13.4 下行通道：调度指令的数据结构

调度指令从总控下发到指定列车的车载系统。每条指令包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| trainId | string | 目标列车标识（如 T1） |
| commandType | string | 指令类型枚举 |
| targetValue | number | 指令参数（具体含义依指令类型而定） |
| reason | string | 下发原因说明（人工可读） |
| issuedTime | number | 指令生成时刻（仿真秒） |
| status | string | PENDING → ACKNOWLEDGED → EXECUTING → COMPLETED / REJECTED |
| source | string | "DISPATCH" 表示总控指令 / "MANUAL" 表示人工调度员干预 |

**指令类型与车载系统的响应行为**：

| 指令类型 | targetValue含义 | 车载系统响应 |
|---------|-----------------|-------------|
| SPEED_UP | 恢复等级 0-3 | 调整ATO巡航速度：在原运行等级巡航速度上叠加对应的速度增量（+0/+5/+8/+10 km/h）。同时按等级缩短计划停站时间（-15/-20/-25/-30s） |
| SLOW | 限速值（km/h） | 将列车临时最高限速降为指令指定的值。在速度降到目标值以下后进入新的稳态巡航。限速在到达下一站后自动清除（除非总控下发新SLOW指令） |
| HOLD | 延长停站秒数（0=调度员手动解除） | 在站停阶段延长停靠时间。车载达到计划停站时间后不立即发车，继续等待HOLD解除 |
| SKIP_STATION | 要甩掉的站台索引（可选，0=甩下一站） | 接近下一站时不制动，以当前巡航速度通过站台。通过后跳到再下一站为目标 |
| CHANGE_LEVEL | 运行等级编码（0=NORMAL, 1=ENERGY_SAVE, 2=EXPRESS, 3=SLOW） | 切换车载的运行等级，包括巡航速度、加减速度、停站时间倍率。若切换到EXPRESS级别，同时设置skipNextStation=true |
| SHORT_TURN | 强制折返站台索引 | 列车在该站停靠后不继续前行，而是执行折返，切换方向后反向发车 |
| EMERGENCY_BRAKE | 无（不使用） | 车载系统收到后立即施加最大紧急制动，速度归零后等待总控进一步指令 |
| RESUME_NORMAL | 无（不使用） | 清除所有临时限速和特殊策略，恢复正常运行等级（NORMAL模式的默认参数） |

### 13.5 上行通道：车载状态上报数据结构

车载系统上报自己的运行状态到总控，采用以下结构：

**周期性状态上报（StatusReport）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| trainId | string | 本车标识 |
| timestamp | number | 上报时刻（仿真秒） |
| position | number | 当前绝对位置（米），使用全线统一公里标 |
| speed | number | 当前速度（km/h） |
| acceleration | number | 当前加速度（km/h/s，正值牵引、负值制动） |
| phase | string | 运行阶段：traction / coast / braking / stopped / emergency_brake |
| direction | string | 运行方向：UP / DOWN |
| currentStationIndex | number | 当前所在站索引（-1=区间运行中） |
| nextStationIndex | number | 下一目标站索引 |
| delaySeconds | number | 车载自身计算的晚点偏差（实际到站 vs 计划到站） |
| activeCommand | object | 当前正在执行的调度指令（如有），指令回执 |
| health | object | 车载健康状态：{ atpOk, atoOk, doorStatus, tractionOk, brakeOk } |

**事件即时上报（OnboardEvent）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| trainId | string | 本车标识 |
| eventType | string | 事件类型枚举 |
| timestamp | number | 事件发生时刻（仿真秒） |
| position | number | 事件发生时位置（米） |
| speed | number | 事件发生时速度（km/h） |
| details | object | 事件详细数据（依类型不同） |

**事件类型与总控的响应策略**：

| 事件类型 | 触发条件 | 上报内容 | 总控响应策略 |
|---------|---------|---------|-------------|
| ATP_EB_TRIGGERED | 车载ATP检测到超速或间距不足，触发紧急制动 | 触发原因、触发位置、当前速度 | 立即标记该车为紧急状态。对所有后续列车（跟在该车后方同方向的列车）下发SLOW指令，限制速度不超过25km/h。重新评估全线晚点传播 |
| STATION_ARRIVAL | 列车到达站台、速度归零 | 到站站台索引、到站时刻、停车位置 | 用实际到站时刻与计划时刻表比对，更新该车的delaySeconds。如果偏差超过60秒触发晚点事件 |
| STATION_DEPARTURE | 列车完成上下客关闭车门后发车 | 发车站台索引、发车时刻、实际停站秒数 | 更新列车状态为DEPARTING。检查该车是否有SPEED_UP指令活跃，如有则在运动学中继续执行提速 |
| STOP_PRECISION_WARNING | 停站偏差超过0.5米 | 目标停车点、实际停车点、偏差距离 | 记录到事件日志。如果偏差超过3米（可能影响车门与屏蔽门对齐），标记为"停站故障"并建议调度员关注 |
| DOOR_FAULT | 车门未正常关闭 | 故障车门编号 | 标记该车为"带故障运行"。调度员可选择在下一终端站安排该车回库 |
| OVERSHOOT_STATION | 列车越过站台未停车（制动失败或甩站信号异常） | 越过站台索引、越过距离 | 立即给该车下发EMERGENCY_BRAKE；通知前方站台人员；对后车下发SLOW指令 |
| PASSENGER_ALARM | 客室紧急报警被触发 | 报警来源车厢编号 | 通知调度员。根据运营规程，通常安排列车在下一站停靠处理（不下发甩站指令） |
| SPEED_LIMIT_VIOLATION | 车载检测到超过当前限速 | 当前速度、限速值 | 如果构成危险（超速10%以上），下发SLOW指令降速；记录为安全违规事件 |

### 13.6 数据一致性：双状态模型

这是系统设计中的一个核心问题。总控系统和车载系统各自维护了一份关于同一列车的状态：

- **总控侧（调度视图）**：多列车简化运动学模型，70km/h巡航、三段式运动（加速-巡航-制动），它不需要知道列车每一秒的具体牵引力变化——它只需要知道列车的大致位置、速度、预计到站时间，以及是否符合计划时刻表。

- **车载侧（真实物理视图）**：单列车精确纵向动力学模型，使用Davis方程计算运行阻力（含v²项），考虑坡度阻力，采用四阶段（traction-coast-braking-stopped）运行、小步长（0.025s）数值积分，输出含真实运动学特征的速度-位置曲线。

**双状态模型的协调机制**：

1. **总控侧是"参考"**：总控根据简化运动学推算列车在哪、多快、预计何时到站。它用于全局调度决策（晚点检测、传播控制、车头时距计算）。

2. **车载侧是"事实"**：车载上报的实际状态覆盖总控侧的推算。当总控收到车载的StatusReport后，将总控侧的TrainState.positionMeters、speed、acceleration更新为车载侧的上报值。如果车载侧的位置与总控侧的推算偏差超过100米，或者时间偏差超过30秒，说明时刻表可能不准或列车遇到了未建模的异常——此时总控触发一次全量晚点重评估。

3. **融合策略**：总控不盲信车载数据（可能延迟或丢失），也不盲信自己的推算（未建模因素导致误差）。采用"信任最新车载数据，但在超时后回退到推算"的策略：如果某个列车在3秒（6个仿真步长）内没有收到新的StatusReport，总控切回使用自己的运动学模型推进该列车。

### 13.7 协同工作流程

以下描述一个完整的"观察-决策-执行-反馈"循环：

**第一步：总控全局态势感知**

总控每500ms轮询一次，从所有列车获得最新的StatusReport。同时维护自己的简化运动学模型，一旦某列车超过3秒未上报，自动回退到推算模式。总控掌握了全线所有列车的位置、速度、方向和预计到站时间。

**第二步：总控晚点评估与调度决策**

总控每30仿真秒执行一次全量晚点评估（evaluateAllDelays）。对于晚点超过60秒的列车，判定恢复等级（L1-L4）并生成SPEED_UP指令。同时执行晚点传播评估（evaluateDelayPropagation），根据前后车间距、前车晚点和恢复潜力，决定对后车是轻度限速还是强力减速。这些指令被暂存在CommandBus中，等待各车载系统拉取。

**第三步：指令下发到车载**

各车载系统在下一次轮询（500ms内）拉取到属于自己的待处理指令。车载的指令解析模块接收到SPEED_UP指令后，提取恢复等级（targetValue 0-3），查询恢复参数表：获取巡航速度增量（0/5/8/10 km/h）和停站缩短量（15/20/25/30s）。将这些参数注入本地ATO控制器，随后的巡航段以更高速度运行，站停阶段以更短时间停靠。

如果收到的SLOW指令，车载设置临时最高限速，本地控制器保证在任何情况下速度不超过此限值。

**第四步：车载执行与实时调整**

车载系统按照自己的纵向动力学模型运行（含Davis阻力、坡度阻力、四阶段状态机）。在每个采样步长（如0.5s），车载计算出实时的位置、速度、加速度、阻力消耗、牵引力需求。如果在下一站接近时检测到stationApproach条件（到站台距离 ≤ 制动距离 × 安全系数），自动切入制动阶段；如果skipNextStation为True，则持续巡航通过该站。

**第五步：状态上报与事件通知**

车载系统每500ms上报一份StatusReport到总控，包含当前position、speed、acceleration、phase、delaySeconds。总控用这些数据更新调度视图，确保后续的晚点传播评估和车头时距计算基于真实数据。如果车载检测到紧急事件（ATP超速触发、停站偏差过大），不等轮询周期，立即推送OnboardEvent到总控。

**第六步：总控根据反馈重新调整**

总控收到车载的异常事件后，将其纳入下一次全量晚点评估（全量评估在收到事件后不等30秒周期，可以提前触发）。例如，如果T3上报ATP紧急制动事件，总控立即：
- 标记T3状态为紧急制动中
- 对T4（T3后方的列车）下发SLOW指令，限速25km/h
- 评估T3的晚点会如何传播到T4、T5
- 如果T3预计晚点超过360秒（L4 CRITICAL级别），对T4施加HOLD在最近的车站扣车等待

**第七步：指令回执与生命周期管理**

当车载系统收到一条指令后，先回复ACKNOWLEDGED状态（已确认收到）；执行完毕后回复COMPLETED（已完成）。如果指令因为车辆自身状态无法执行（例如：SLOW指令要求限速到25km/h，但列车当前的制动减速度不够，无法在目标位置前减速到25km/h），车载回复REJECTED并附上原因。

总控端维护每条指令的状态机：PENDING（待拉取）→ ACKNOWLEDGED（已下发）→ EXECUTING（执行中）→ COMPLETED / REJECTED（终态）。已完成的SPEED_UP指令在列车晚点恢复到15秒以下后自动失效。

### 13.8 车载侧需要新增的API接口

为实现上述协同交互，车载系统需要新增以下接口（调用总控侧）：

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/dispatch/commands/{trainId} | GET | 拉取该列车当前待处理的调度指令（状态为PENDING的指令链表） |
| /api/dispatch/report/status | POST | 推送周期性状态上报（StatusReport） |
| /api/dispatch/report/event | POST | 推送即时事件上报（OnboardEvent） |

### 13.9 总控侧需要新增的接口与逻辑

总控侧需要新增以下能力：

**指令管理模块（CommandBus）**

在总控后端新增一个指令队列管理器，功能如下：
- 为每列车维护一个指令队列（FIFO，最多保留5条PENDING指令）
- 车载拉取时返回该列车最新的PENDING指令
- 收到ACK后状态变更为ACKNOWLEDGED
- 如果3秒内未收到ACK，自动重发
- 收到COMPLETED或REJECTED后标记为终态并归档到commandLog

**状态接收与融合模块（StatusFusion）**

总控新增一个状态融合服务，功能如下：
- 接收车载StatusReport，更新总控侧该列车的TrainState（覆盖position、speed、acceleration、status、delaySeconds）
- 统计每列车的最近上报时间戳
- 对超过3秒未上报的列车，将flag标记为"stale"，调度决策时对这些列车仅使用保守策略（不允许尝试激进的赶点，下发SLOW时留更大余量）
- 当车载位置与总控推算位置偏差超过100米时，触发位置重对齐：以车载位置为准，同步更新总控侧的stationIndex等派生字段

**异常事件处理模块**

总控新增事件处理器，针对不同的OnboardEvent执行对应的响应策略（见13.5节事件类型表）。关键设计是：收到紧急事件（如ATP_EB_TRIGGERED）后，总控不等30秒周期，立即执行一次针对性的调度重评估——仅重新评估受影响的相关列车（事件列车及其后方列车），而不是全量重评估所有列车，以降低计算开销。

### 13.10 车载侧逻辑改造要点

车载系统需要从"独立仿真播放器"改造为"在线实时控制器"，核心改动包括：

1. **从"预计算全部曲线一次返回"改为"实时逐帧计算"**：当前的车载仿真（VehicleSimulationService.run）是一次性计算完从A站到B站的完整状态曲线后返回。在线模式下，需要改为逐仿真步长计算——每次只计算从当前时刻到当前时刻+步长时间的状态增量，保持内部积分器状态（当前速度、位置、加速度）而非每次都从零开始。

2. **增加指令监听线程**：周期性地从总控拉取指令。收到新指令后不中断当前计算，而是在下一个积分步开始时应用新的参数（巡航速度、限速值、停站时间等）。

3. **增加状态上报逻辑**：每完成一次积分步（如500ms的对外采样间隔），将当前状态组装成StatusReport并推送到总控。

4. **增加安全事件检测与上报**：在每一个内部子步（如0.025s）中检查：是否超过限速、制动减速度是否异常、车门状态是否正常等。检测到异常后立即生成OnboardEvent并推送。

5. **模块化指令解析**：新增CommandParser模块，将总控下发的调度指令映射为车载本地控制参数：
   - SPEED_UP（targetValue=recoveryLevel）→ 巡航速度 += RECOVERY_SPEED_BOOST[level]、dwell时间 -= RECOVERY_DWELL_CUTS[level]
   - SLOW（targetValue=limitSpeed）→ maxSpeedLimit = limitSpeed
   - HOLD → 延长dwell剩余时间
   - SKIP_STATION → skipNextStation = true（在接近下一站时跳过制动阶段）
