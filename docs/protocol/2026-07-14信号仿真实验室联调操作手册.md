# 信号仿真实验室联调操作手册

适用范围：本仓库的信号仿真后端、控制中心、车载页面和实验室司机台。目标是完成一列车的正向站间流程，而不是在未验证的接口上做全设备联动。

## 已实现的安全链路

```text
控制中心办理完整的站间进路链并原子绑定列车
  -> 信号周期刷新区段占用、联锁状态和 MA
  -> 满足条件才给司机台 ATO_READY
  -> 司机台实体双 ATO 按钮互锁后上报 ATO_START
  -> 后端 ATO 运行；同一状态同步给调度、车载和 HIL 屏
  -> 到站停车，开门、关门
  -> 新进路和新 MA，重复下一站流程
```

ATS 只能提交调度意图，不能直接放行司机台。网页的 `DEPART`、`depart-confirm`、`sync-state.departureConfirmed` 都不能取得 ATO 授权。联锁撤销、MA 不足、门开、ATP 紧急制动都会撤销授权；ATO_READY 本身不产生运动。

实验列车被 704 桥接器注册后，调度控制模块只保留状态融合、时刻表监视、联锁和 MA 计算，不再生成、执行或接受该列车的 `DEPART` 指令；任何离站都只能由满足全部联锁条件后的 PLC `ATO_START` 新上升沿触发。重置 704 会话后才会释放这一司机台执行权。

## 接口基线

| 设备 | 地址和方向 | 实际帧 | 本次结论 |
|---|---|---:|---|
| 司机台 PLC | `192.168.100.123:8001`，PLC -> 后端 | TCP 46 B，约 100 ms | 可以作为主输入链路；`8002/8003` 只镜像诊断，默认不控制列车 |
| 网络屏 HMI | `192.168.100.121:8888`，后端 -> 屏 | TCP 570 B | 真实抓包中 1140 B 是两个粘连的 570 B 帧；老师允许后才开启 |
| 信号屏 MMI | `192.168.100.122:9999`，后端 -> 屏 | TCP 68 B | 报文头仍保留历史 `totalLen=62`；老师允许后才开启 |
| 视景 | `192.168.100.102 -> .124:8303` | 单向 UDP 128 B | Vision 1.3：`77` 个信号机、`29` 个道岔和 `16 B` 本车字段 |

视景系统以 `128 B` 为唯一协议基线，配置固定为 `hil.vision.profile=documented-128`。UDP `send()` 成功只表示操作系统已投递，真实画面是否接收仍需在 `.124` 设备端或现场抓包确认。

PLC 回写在普通模式下默认关闭；`scripts/start-platform.ps1 -Lab` 会启用抓包对应的 28 B / `dataLen=4` 格式。历史配置 `vehicle.protocol704.plc-output-enabled` 已不再产生写帧；只有 `LabHilGateway` 可以通过已建立的 PLC 连接发送回写，避免两个周期任务向同一设备交错写入。

## 联调前准备

1. 由老师指定本机 `192.168.100.0/24` 地址、掩码和网关。不要自行占用 `.102`、`.121`、`.122`、`.123` 或 `.124`。
2. 确认没有同时运行抓包脚本、旧 704 客户端、同学程序或第二个本系统后端。一个 PLC 端口只允许一个客户端连接。
3. 保持默认关闭所有物理输出，启动后端：

```powershell
cd E:\大三下小学期\SemesterTraining-III\backend
mvn spring-boot:run
```

4. 启动控制中心和车载页面。确认后端可访问 `http://localhost:8080/api/signal/status`。
5. 只读检查 PLC 连通性：

```powershell
Test-NetConnection 192.168.100.123 -Port 8001
```

在没有老师许可前，不要对屏幕端口或 PLC 发测试字节。

## 一列车正向站间流程

以下流程先限制为平台站起步的正向运行。当前后端没有把任意 `LK_ID/Seg` 自动转换为物理里程，也未实现实物折返方向切换；不要宣称这两项已经可用。

在当前教师静态数据中，站台中心公里标为站 1 `K0+313.000`、站 2 `K1+660.520`，因此站 1 至站 2 的相对运行距离是 `1347.520 m`，下一站绝对目标为 `K1+660.520`。站台公里标与 Seg 图的推导里程尚未有经确认的一一映射。后端不会把单条站场进路错误当作完整站间授权，而是从源、目标同向站台 Seg 和 CBI 进路有向图解析唯一的最短进路链。首段站 1 到站 2 的已验证序列是 `R9(F5-Z4) -> R28(Z4-SC4)`；站 2 到站 3 是 `R29(SC4-SC)`。只有整条链建立，MA 才会投影至下一站平台公里标；桥接器确认停准目标站后才整链释放。关门本身不会重新授权。

当前适配只支持单车、正向、相邻站间演示。在线路信号页面新增实验列车时，可分别选择起点和终点；终点下拉只显示从该起点连续、验证可到达的站点。即使绕过网页提交硬件控制请求，后端也会逐站复核。依据教师 CBI 静态数据，站 1 至站 13 的正线序列已连续验证：`1->2 R9/R28`、`2->3 R29`、`3->4 R36`、`4->5 R37/R38`、`5->6 R39/R48`、`6->7 R49/R50`、`7->8 R51/R53/R55`、`8->9 R65`、`9->10 R66/R67/R72`、`10->11 R73/R74`、`11->12 R75/R77`、`12->13 R89`。解析时会合并起止信号、计轴区段、保护区和联锁区均相同的重复 Route 行，并在同长度候选中只接受使用最少反位道岔 Seg 的唯一正线路径；仍有并列时继续拒绝。该适配不是 `LK_ID/Seg` 精确定位或真实区段占用的替代品。

1. 在控制中心新增列车，选择起始站、正向 `UP` 和唯一的 `trainId`，例如 `T1`。`headLinkId` 仅作记录；先不要把它当作精确定位输入。
2. 在车载页面对同一个 `trainId` 启动硬件控制仿真：`fromStationId`、`toStationId` 选择相邻正向站，`hardwareControlEnabled=true`。此时应为 `READY_TO_DEPART`，不是自动发车。
3. 调用或点击 704 连接，并确认 `T1` 收到有效 46 B 帧。检查 `GET /api/vehicle/protocol704/status?trainId=T1`：控制端口应为 `8001`，且 `simulationReady=true`。
4. 司机台先执行老师规定的准备动作：方向手柄零位、主手柄零位；钥匙右/开、关、再开。检查 `GET /api/vehicle/protocol704/ato-workflow?trainId=T1`，钥匙状态不能是 `KEY_SWITCH_OFF`。完成钥匙动作后，办理或等待 ATO 时必须将方向手柄置于前进、主手柄置于零位；这两个 PLC 状态也是 ATO 灯的必要条件。
5. 后端信号周期会为实验列车自动办理当前站至下一站进路，原子建立完整站间链并绑定 `T1`。等待“实验列车”栏显示 ATO READY；“办理下一站进路”按钮仅保留为自动办理失败后的人工重试入口。不得用普通单条进路办理代替，也不得先绿灯后绑定。
6. 后端立即重算信号周期。检查 `GET /api/signal/driver-workflow?trainId=T1`：
   - `route.state=ESTABLISHED`；
   - `movementAuthority.endOfAuthorityM` 覆盖下一站台，且 `maxSpeedKmh>0`；
   - 钥匙为开、方向手柄为前进、主手柄为零位、`atoReady=true`；
   - `workflow=ATO_READY`；
   - `atoReadinessBlockingReason=NONE`。
7. 若已经获得老师批准 PLC 灯回写，才打开该受控阶段，并观察司机台 ATO 灯闪烁。没有这个批准时，API 的 `atoReady=true` 是后端状态，不代表实体灯已经亮。
8. 初次联调可按老师手册低级位手动牵引、速度低于 `10 km/h`，随后回到零位；这一步必须已经有第 5 至 6 步的进路和 MA，不能在无 MA 下测试牵引。
9. 主手柄零位、方向前进，按住实体两个 ATO 启动按钮约两秒。后端只接收 PLC 的最终 `byte34 bit7=ATO_START`，不会在软件中模拟“两按钮两秒”。该位必须是取得授权后的新上升沿：若在 `ATO_READY` 前已按住，或一直长按未松开，后端会拒绝 `ATO_START_NOT_NEW_EDGE`；松开两个按钮后重新按下。状态应由 `ATO_READY` 变为 `ATO_RUNNING`。
10. 到下一站后，ATO 自动停车；仅对线路信号页创建的实验列车，后端自动开右侧门、驻留约 5 秒并关门。关门后仍保持禁止发车，等待旧进路释放以及下一站完整进路和 MA 重新建立。
11. 新授权建立且钥匙保持开、方向保持前进、主手柄保持零位后，后端自动启动下一段 ATO，无需再次按实体双按钮。起点站第一次启动仍必须人工完成钥匙循环和双按钮操作；终点站只自动完成门循环，任何情况下都不再次发车。自动办理失败时可使用“办理下一站进路”人工重试。

## 现场停止条件

- 46 B 长度、协议头或动作字段和当前映射不一致：主手柄回零，断开 704，保存抓包，不改业务映射硬凑。
- `workflow=WAITING_SIGNAL_AUTHORIZATION`：检查绑定进路、MA、门状态；不要用网页确认按钮绕过。
- `atoReadinessBlockingReason=DIRECTION_NOT_FORWARD` 或 `MASTER_HANDLE_NOT_ZERO`：按老师手册将方向手柄置于前进、主手柄归零，等待下一帧 PLC 状态；不要在灯未闪烁时长按 ATO 按钮。
- `atoReadinessBlockingReason=MA_DOES_NOT_REACH_NEXT_STATION`：MA 没有覆盖桥接器的下一站台目标，检查完整站间进路链是否全部建立；不得以任意单条 `Rxx` 代替。
- `workflow=ATP_EMERGENCY_BRAKE`：保持停车，复盘 MA、占用、定位和速度数据；不得自动恢复牵引。
- Vision 帧不是 `128 B`、PLC 灯无实物回显：关闭对应输出并保留日志，不要现场临时修改包长或字段偏移。

## 运行中交互抓包

平台和 704 后端已经连接 PLC 时，不要再运行会主动连接 PLC 的
`plc704_capture.py`。请在管理员 PowerShell 中运行旁路抓包脚本：

```powershell
cd E:\大三下小学期\SemesterTraining-III
python scripts/capture_lab_session.py --train-id T1 --label baseline
```

每次操作前输入标签并回车，或使用脚本显示的 `1` 至 `12` ATO 快捷步骤；
`/status` 显示最新的 704、ATO、信号和 HIL 状态，`/stop` 停止并保存。
结果写入 `captures/lab-session-<时间>/`，其中 `traffic.pcapng` 可用
Wireshark 打开，`operator-events.csv` 用于按时间对齐实体操作，
`runtime-status.jsonl` 用于核对当时的 ATO 授权、拒绝原因、速度和位置。
该脚本只使用 Windows PktMon 监听网卡，不会打开任何实验室设备连接。

## 受控扩展顺序

完成主链路后，每次只打开一个通道，并通过 `GET /api/hil/status` 确认帧数和字节数：先 MMI `68 B`，再 HMI `570 B`。PLC 灯回写、Vision 128 B 和真实终端折返均不是本流程的前置条件。

真正终端折返需要补齐反向线路物理模型、实际方向上报及折返互锁验证；任意 `LK_ID/Seg` 起车需要补齐 `LK_ID + offset -> LineProfile.locateMileage()` 的经确认映射。二者在完成前保持禁用。
