## 3. PLC（.123）— 我们已基本打通「读」

### 3.1 PLC → 上位机（46B，100ms）

与你们已抓到的帧一致，仓库 `Protocol704FrameParser` 已覆盖主手柄闭环：

| 偏移 | 协议字段 | 本仓库 |
|------|----------|--------|
| 0–7 | magic / len | ✅ 校验 |
| 8–19 | 时间 | ✅ 解析 |
| 20–23 | verify | ⚠️ 只存，不校验 |
| 24–25 | 指示灯/模式具备位 | ⚠️ 解析，未联锁 |
| 26–27 | 车辆速度（WORD） | ⚠️ 文档称上位机回写速度；朋友采集未证明 PLC 实际回显或单位 |
| 28–43 | 按钮/手柄/级位 | ✅ 映射 traction/brake/coast/EB |
| 44–45 | 预留 | ⚠️ |

### 3.2 上位机 → PLC（26B 文档格式；28B 朋友候选格式）

老师协议通信总表明确写 `26B` / `dataLen=2`；字段表却列出 offset26 的速度 WORD，
自身矛盾。朋友系统的真实采集记录了其写出的 138 个 `28B` / `dataLen=4` 帧，但紧随
其后的 PLC 46B 输入速度均保持 0，故该记录**不能证明 PLC 已接受 28B**。  
**字段意图**：回写指示灯 + **车辆速度**（PLC 再在 46B 的 offset26 回显）。

**本仓库：已实现两种编码与同连接写出，但默认关闭**。默认
`hil.plc-output.frame-format=documented-26`；`capture-variant-28` 仅用于老师允许的
现场对照。启用后网关每 500ms 向已连接 socket 回写一帧。先验证灯态和 PLC 46B 回显，避免与采集工具并发。

## 4. 信号屏（实测：.122:9999，66B）

我们 → 屏，周期推送。关键字段与仿真映射：

| 协议字段 | 建议数据源（已有） |
|----------|-------------------|
| 时间 | 系统时钟 |
| `_nCurr/Next/EndStationID` | 仿真 `from/to/nextStationId` |
| `_nSpeed` FLOAT | `TrainState.velocity` (m/s) |
| `_fAcceleration` FLOAT | `TrainState.acceleration` |
| `_fSpeedLimit` | `LineProfile.speedLimit` 或 MA `maxSpeedKmh` |
| `_nMode` | `DrivingMode` → 协议 DTO/ATO/AR/SM/RM（需对照表） |
| `_nPullState/_nBrakeState/_nUrgencyStopState` | 704 主手柄映射 / EB |
| `_fNextStationDist` | `target - position` |
| `_nSigState` | `SignalAspect.toDriverConsoleScreen()`（已有 红/绿/白） |
| `_nRunDir` | `Direction.fromDriverConsole`（0 上 / 1 下） |

**本仓库**：MA/`SignalAspect`/`Direction` 已对齐协议语义；`LabHilGateway` 已可周期写
66B（默认关闭）。本次采集确认发送节奏和连通性，尚需逐字段核对实体屏显示。

## 5. 网络屏（实测：.121:8888，572B）

大屏车辆 HMI。需要速度/加速度/牵引力/网压/门状态/制动缸压力等。

| | |
|--|--|
| 我们 → 屏 | 572B 周期帧 |
| 屏 → 我们 | 26B 牵引切除（bit0–5 = 1–6 车） |

**本仓库**：`LabHilGateway` 已可周期写 572B（默认关闭）；2026-07-12 采集到 1,183
次 572B 发送和同数量 572B 回显。网络屏反向 26B 牵引切除尚未实现或验收。

仿真侧已有：速度、加速度、牵引/制动级位、模式、站 ID；大量电气量（制动缸压力、逆变器…）本仿真未建模 → 可先填 0 / 默认值（协议也允许「没值显示 0」）。

## 6. 视景（你方 .124；文档旧 IP 18.32.115.x）

UDP 包 `TCMS2VIEW`：信号机 77 + 道岔 29 + 本车速度(mm/s) + 位置(mm) + 边号 + 方向 + 他车列表。

**本仓库**：`VisionProtocolAdapter` 是 JSON 占位；Web 3D 是自研前端，**不等于**老师视景 UDP。

要用老师 3D：需确认 `.124` 的收发端口，再按 Version1.3 表打包。

## 9. 你现在可以怎么「使用」

### 路径 A — 今天就能做（只读 PLC 控仿真）

1. `ping 192.168.100.123` 通  
2. `python scripts/tcp704_live_demo.py --reconnect` 看手柄摘要  
3. 后端 + 车载前端：先跑仿真（记住 `trainId`）→ Protocol704 **Connect**  
4. 推方向/牵引/制动 → 看仿真速度是否变  

这里的关键一致性是：车载页面已上线的 `trainId` = Protocol704 面板 Connect 的
`trainId`。`hil.train-id` 仅在开启信号屏、网络屏或 PLC 状态回写等 HIL **输出**时需要
配成同一列车；它不影响 PLC 输入控制绑定。

### 路径 B — 下一步完善（建议顺序）

1. **逐项核验信号屏 66B**：实体屏先核对速度、模式、限速。  
2. **PLC 回写验证**：先按 26B 发送 0 值灯态；记录 PLC 46B 回显，再由老师允许时对比 28B。  
3. **逐项核验网络屏 572B**：先核对核心字段，其余默认值后补建模。  
4. **视景 UDP**：确认 .124 端口后再做。  

### 路径 C — 现场必问老师的 3 个问题

1. 视景 `.124` 的 UDP 端口是否仍是 8302/8303？边号是否用附件 2 的 48 边？  
2. 网络屏反向 26B 牵引切除帧是否在该型号设备启用、其有效确认机制是什么？  
3. PLC 回写是否要求某个固定发送时序或校验值，而非当前采集到的工作样本？

## 10. 协议文档内部注意点

1. **信号机编码有三套**：信号系统 3.3.2 / 视景 ES bit / 视景控制机内部色值 — 勿混用。  
2. **网络屏文档里 `_nSpeed` 偏移写了两次**（一次 42 与站字段重叠）— 实现时以「总长 66 / 572」和老师抓包为准。  
3. **PLC 入站 magic**：线上为 `55 AA 55 AA`（小端 DWORD=`0xAA55AA55`），与现解析一致。  
4. **列车 1 = 真车**（协议「其他」节）；其余为仿真车。
