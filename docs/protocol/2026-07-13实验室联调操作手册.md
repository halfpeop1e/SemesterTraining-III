# 704 司机台与本系统联调操作手册

> 已由 [2026-07-14信号仿真实验室联调操作手册](2026-07-14信号仿真实验室联调操作手册.md) 取代。现场请以新版的联锁/MA/ATO 流程和 `46B / 570B / 68B / 128B` 协议基线为准；本文保留作历史记录。

适用场景：明天在实验室接入 `192.168.100.0/24` 网段，用老师的司机台输入控制本仓库后端中的仿真列车。本文只覆盖已经有实测证据的 PLC 输入链路，并把设备输出放在最后的受控验证阶段。

## 一句话结论

明天可以验证的主链路是：

```text
老师司机台的钥匙/方向/主手柄/级位
  -> 192.168.100.123:8001 的 PLC 46B 帧
  -> 本系统 Protocol704 解析与控制桥
  -> 已上线的同一 trainId 仿真列车
  -> 本系统车载页面、列车速度和位置变化
```

不需要额外创建另一辆“实验列车”。车载页面对目标车点击“上线并等待发车”后，后端会为这个 `trainId` 创建可接收 PLC 控制的仿真上下文。704 面板连接时必须填写同一个 `trainId`。没有先上线时，后端会显示 `NO_ACTIVE_SIMULATION`，此时司机台输入会被拒绝而不会错误驱动车辆。

朋友的采集数据确实包含真实设备发来的 PLC 46B 帧，能证明司机台输入读取方向、牵引、钥匙、ATO 等字段是可行的；但朋友软件发往 PLC 的 28B 帧没有得到 PLC 速度回显，不能据此认定 PLC 回写已经验收。因此本次先不启用任何输出。

## 已知接口状态

| 通道 | 地址 | 现状 | 明天的动作 |
|---|---|---|---|
| 司机台 PLC 输入 | `.123:8001`，可备选 `8002/8003` | 已由真实 46B 记录验证；本系统已解析并接入车辆控制桥 | 先只读采集，再让后端连接控制 |
| 网络屏 | `.121:8888` | 572B 发送与同长度回显有现场样本；字段显示待逐项验收 | 主链路通过后，老师同意才开 |
| 信号屏 | `.122:9999` | 66B 发送有现场样本；没有回包，不代表显示字段已验收 | 主链路通过后，老师同意才开 |
| 视景 | `.124` UDP | 端口、边号、信号机排列未确认 | 本次不启用 |
| PLC 回写灯/速度 | `.123:800x` | 协议 26B 与朋友 28B 记录相互矛盾，均未获实体验收 | 本次不启用 |

> 文档中 `.121/.122` 的名称有对调。以实测端口为准：`.121:8888` 是网络屏，`.122:9999` 是信号屏。

## 今晚在宿舍完成的检查

不接实验室网也可以执行：

```powershell
python scripts/plc704_capture.py --self-test
python scripts/plc704_capture.py --replay hil-704-capture-1783844035066-1783844174581/records.jsonl
python -m py_compile scripts/plc704_capture.py scripts/lab_start_train.py
cd backend
mvn -q test
```

第二条会用朋友采集的真实 PLC 帧重新解析，并产生 `captures/plc704-replay-<时间>/`。检查其中的：

- `summary.json`：应看到非零的 `frames_valid`，平均帧间隔约 100ms。
- `frames.csv`：每帧的钥匙、方向、主手柄、牵引/制动级位及原始十六进制。
- `events.jsonl`：字段发生变化时的前后值。

不要在宿舍运行不带 `--replay` 的采集命令，因为默认 PLC 地址只在实验室可用。

## 到实验室前的网络准备

1. 向老师确认本机要固定为 `192.168.100.12`、子网掩码、网关是否按实验室要求配置。不要自行设置与现有设备冲突的 IP。
2. 使用有线网卡接入；临时关闭 VPN、代理和会改路由的虚拟网卡。无线网可以保留，但不能成为 `192.168.100.0/24` 的优先路由。
3. 进入网段后先执行：

```powershell
ipconfig
ping 192.168.100.123
Test-NetConnection 192.168.100.123 -Port 8001
Test-NetConnection 192.168.100.121 -Port 8888
Test-NetConnection 192.168.100.122 -Port 9999
```

预期是 `.123:8001` 可达。屏幕端口是否可达只说明 TCP 服务在线，不等于允许发送或字段已正确显示。

## 阶段 A：先做只读数据采集

这个阶段的目的只是确认老师司机台的真实按钮和手柄是否按预期进入 46B 帧，不能同时连接本系统的 704 后端。

1. 确认后端 704 已断开，且没有正在运行的 `tcp704_live_demo.py`、`lab_start_train.py` 或其他 PLC 客户端。
2. 在项目根目录运行 2 分钟的只读采集：

```powershell
python scripts/plc704_capture.py --duration 120 --interactive-label --reconnect --label baseline
```

3. 控制台出现 `connected; receiving only` 后，先保持 3 秒不动，输入 `baseline` 后回车。
4. 一次只操作一个项目，每个动作保持约 2 秒、回零约 2 秒。**EB 只在老师明确允许、且列车处于安全静止状态时测试。** 建议顺序：

   - 钥匙：关 -> 开 -> 关
   - 方向手柄：零位 -> 前进 -> 零位；如允许再测后退
   - 主手柄：惰行 -> 牵引，分别尝试 20%、50%、100% -> 惰行
   - 制动：惰行 -> 制动，尝试 20%、50% -> 惰行
   - 确认、ATO 启动、模式升级/降级、开左门、开右门、关门
   - EB：仅老师允许时单独做一次，并立即释放

   每次操作前在采集窗口输入标签，例如 `traction_20`、`door_left`、`ato_start`，再回车。标签会写入 `events.jsonl`。
5. 采集结束后，保留终端打印的 `captures/plc704-capture-<时间>` 路径。检查 `events.jsonl` 是否能看到操作对应字段变化。没有变化时，先判断是否连错 PLC 端口或设备没有把该输入送到本 TCP 通道，不能直接改业务映射。

`plc704_capture.py` 只使用 TCP `connect` 和 `recv`，没有任何 `send`、`sendall`、PLC 回写或屏幕写入代码。它可以安全用于本阶段，但每个 PLC 端口仍应只让一个客户端连接。

## 阶段 B：让司机台控制本系统列车

只有阶段 A 确认有稳定 46B 帧后，关闭采集工具，才进入此阶段。

1. 从阶段 A 的终端按 `Ctrl+C` 退出，确认进程已经结束。
2. 以默认安全配置启动后端。`application.yml` 中 `hil.enabled`、`signal-screen.enabled`、`network-screen.enabled` 和 `plc-output.enabled` 都必须保持 `false`。

```powershell
cd backend
mvn spring-boot:run
```

3. 启动车载前端，选定一辆目标仿真列车，点击“上线并等待发车”。记下页面的 `trainId`，例如 `LAB1`。然后确认 704 面板显示“仿真已准备”。
4. 在该面板点击“连接”，并确认连接使用的 `trainId` 与刚刚上线的完全相同。
5. 观察面板状态，预期：至少一个端口 `connected=true`，随后 `receivedValidFrame=true`；`simulationReady=true`，而非 `NO_ACTIVE_SIMULATION`。
6. 在司机台按以下顺序操作：钥匙开 -> 方向前进 -> 主手柄牵引低级位。若业务状态要求，按一次确认键。观察后端 704 面板中的 `lastMappedCommand`、`lastCommandLifecycle`，以及列车 `velocityMs` 和位置是否变化。
7. 验收标准不是“TCP 连接成功”，而是同时满足：

   - 收到稳定 46B 帧；
   - 映射命令与手柄一致，例如牵引是 `traction`，惰行是 `coast`；
   - 生命周期显示 `EXECUTED` 而非拒绝原因；
   - 同一个 `trainId` 的速度或位置发生合理变化；
   - 松手到惰行或制动后，列车状态相应变化。

可用脚本快速准备本地仿真上下文，但在第一次实体操作时更建议通过页面完成，便于你看到 `trainId` 和状态：

```powershell
python scripts/lab_start_train.py --train-id LAB1 --skip-connect
```

这个命令只验证后端仿真与测试帧，不连接实体 PLC。真正连接仍在车载页面的 704 面板完成。

## 常见问题与停止条件

| 现象 | 首先检查 | 不应做的事 |
|---|---|---|
| `Test-NetConnection` 失败 | 本机 IP、网线、VLAN、老师是否允许连接 | 不要扫描全网或反复快速重连 |
| 连上但收不到 46B | 试老师指定的 `8002/8003`，每次只试一个端口 | 不要同时启动三个采集程序 |
| 收到帧但 `NO_ACTIVE_SIMULATION` | 目标车是否已“上线并等待发车”，704 是否同一 `trainId` | 不要为了绕过它随便创建/替换列车 ID |
| `MODE_NOT_MANUAL` / `NOT_READY_TO_DEPART` | 按真实业务流程设置模式、确认、发车条件 | 不要直接改后端安全判断 |
| 牵引命令执行但速度不动 | 是否存在联锁、停车、供电或线路授权限制；记录状态截图 | 不要马上启用 PLC 回写或屏幕输出 |
| PLC 连接频繁断开 | 确保只有一个客户端，检查网线和端口 | 不要并发运行采集工具和后端连接 |

出现未知帧长度、未知主手柄值、指令与动作不匹配、或设备异常时，立即回零并断开本系统的 704 连接；保留采集目录和后端日志，再离线分析。不要在没有老师确认时启用 `hil.plc-output.enabled`，不要使用 `capture-variant-28`，不要尝试视景 `.124` 的 UDP 输出。

## 阶段 C：后续受控验证，不在主链路之前执行

在阶段 B 已经通过、老师允许且场地处于安全状态时，按一次只开一个通道的原则验证：

1. 先信号屏 `.122:9999` 的 66B，核对速度、模式、限速和下一站。
2. 再网络屏 `.121:8888` 的 572B，核对速度、门状态和核心车辆状态。
3. PLC 回写最后验证，默认只准用文档规定的 `documented-26`；只有老师允许且 26B 无效时，才以单独实验对照 28B。每次写入后必须看实体灯态或 PLC 46B 回显，不能以“程序无报错”作为成功。
4. `.124` 视景必须先向老师确认 UDP 端口、边号、信号机顺序和版本，当前不要发送。
